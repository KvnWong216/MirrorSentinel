# Sentinel-LIO 系统架构报告

## 1. 项目定位

本项目面向镜面、反光面等高退化场景下的移动感知与建图问题，整体上以 LiDAR-Inertial Odometry 为主干，在保持 FAST-LIO 类系统高频、轻量、稳定优势的基础上，引入视觉深度与镜面区域感知能力，对反射导致的伪点、鬼影和几何污染进行抑制。

从仓库组织和运行入口来看，这不是一个单一算法脚本仓库，而是一个完整的研究型系统工程，包含以下三条主线：

1. 在线运行主线：以 `sentinel_lio` 为核心，完成 LiDAR-IMU 预处理、状态估计、地图维护和结果发布。
2. 感知增强主线：以 `mirrorsentinel_visual_prior` 为核心，提供 DA3 单目深度先验，并为镜面/玻璃区域筛查提供视觉侧证据。
3. 评测验证主线：以 `evaluation_tools` 和 `baselines` 为核心，形成“方法实现 + 基线对比 + 指标评估”的闭环。

这意味着项目的目标并不只是“跑通一个 SLAM 节点”，而是构建一个可验证、可比较、可扩展的退化场景鲁棒定位建图研究平台。

## 2. 仓库总体结构

仓库主要由四部分组成：

| 目录 | 角色 | 说明 |
| --- | --- | --- |
| `Sentinal-LIO_ws/` | 主工作空间 | 在线系统主体，包含 `sentinel_lio` 与视觉先验 ROS 封装 |
| `baselines/` | 对比方法区 | 收纳 FAST-LIO、LIO-SAM、LVI-SAM、R3LIVE 等基线相关工作区或依赖 |
| `evaluation_tools/` | 评测工具区 | 轨迹记录、批量评估、地图结构指标与结果汇总 |
| `dataset/` | 数据区 | rosbag 与实验数据输入位置 |

其中，`Sentinal-LIO_ws/` 是系统运行核心，`baselines/` 和 `evaluation_tools/` 体现了项目从一开始就面向“研究比较”和“实验复现”而非单次 demo。

## 3. 在线系统分层架构

在线系统可以拆成五个层次：

### 3.1 传感器接入层

这一层负责接入原始传感器流，包括：

1. LiDAR 点云输入
2. IMU 输入
3. 左右目图像输入
4. 视觉生成的深度图与镜面 mask 输入

在当前 Ouster 运行配置中，典型输入主题包括：

1. `/ouster/points`
2. `/ouster/imu`
3. `/zed2/zed_node/left/image_rect_color`
4. `/zed2/zed_node/right/image_rect_color`
5. `/vfm/depth_image`
6. `/vfm/mirror_mask`

这一层的设计重点是保持 ROS 原生通信方式，最大化兼容现有数据采集、录包与回放流程。

### 3.2 时空对齐与预处理层

`sentinel_lio` 中的预处理模块承担两类工作：

1. 点云格式适配
2. LiDAR-IMU 时间组织

系统支持 Livox、Velodyne、Ouster、MARSIM 等不同类型输入，并通过统一预处理接口把异构点云转为后端一致的数据形式。这一层的价值在于把设备差异限制在入口附近，避免主估计器对具体雷达型号产生强耦合。

在时间维度上，系统将 LiDAR 与 IMU 缓冲区组织为可同步的数据包，为后续 IMU 去畸变和迭代滤波提供连续观测序列。也就是说，系统不是把点云和 IMU 当作简单并行流处理，而是明确围绕“扫描周期”构建融合批次。

### 3.3 视觉感知增强层

`mirrorsentinel_visual_prior` 负责把 DA3 深度、可选反射 mask 和目录 mask prior 封装成独立 ROS 节点。其核心职责是：

1. 订阅 RGB 图像
2. 执行 DA3 单目 metric depth 推理
3. 发布 `/vfm/depth_image`
4. 可选发布 learned/reflection directory mask 到 `/vfm/mirror_mask`
5. 通过 ROS 话题向主建图系统提供视觉几何先验

该模块采用单独 Python 节点运行方式，并依赖独立 Conda/DA3 环境。这种组织方式体现出一个非常明确的设计思想：将“高依赖、重运行时环境”的视觉推理链路与“高稳定、低时延”的 LIO 主链路解耦。这样做的好处是：

1. 视觉模型可以独立迭代，不破坏主建图程序的编译与部署逻辑。
2. 推理侧环境管理与 ROS/C++ 主程序隔离，便于迁移和调试。
3. 当视觉模型升级时，只需要替换模型与封装，不需要重构 LIO 主干。

### 3.4 镜面感知融合层

这是项目最具辨识度的部分。系统并没有把视觉替代 LiDAR，而是把视觉作为“辅助判别源”，在 LiDAR 主链路中引入镜面感知模块 `MirrorSentinel`。

这一层承担三件事：

1. 接收视觉深度图与镜面 mask
2. 将 LiDAR 点投影到相机视域
3. 为点云赋予镜面相关置信度，或者为镜面区域提供额外筛查依据

从设计上看，这一层不是另起一套前端，而是插入在“点云去畸变之后、状态更新之前”的关键位置。也就是说，你们保留了 LIO 的状态估计主逻辑，只在观测构造阶段对点的可信度进行重解释。这是一种非常克制且工程上稳妥的融合方式。

它的本质是：不把视觉当作独立里程计，而是把视觉作为 LiDAR 残差形成阶段的可靠性调制器。

### 3.5 状态估计与地图维护层

系统主干仍然是 FAST-LIO 风格的高频 LiDAR-IMU 紧耦合估计：

1. IMU 驱动的前向传播与去畸变
2. 基于局部地图的最近邻搜索
3. 点到面的约束构造
4. 迭代卡尔曼更新
5. ikd-Tree 增量地图维护

这部分体现出项目的核心策略：在不牺牲实时性的前提下，尽可能保留成熟 LIO 主干的稳定性和计算效率。镜面感知不是替代这个主干，而是为其提供更好的观测质量。

地图维护层采用局部视场裁剪与增量更新机制，既保证在线效率，也为中长时间运行提供内存与查询复杂度控制。

## 4. 运行数据流

项目的在线数据流可以概括为：

```text
LiDAR + IMU
  -> 预处理 / 时间组织 / 去畸变
  -> LiDAR 点云主观测
  -> 镜面感知模块根据视觉深度与 mask 赋予点级置信度
  -> 迭代滤波更新状态
  -> 增量地图维护
  -> 发布 odometry / path / registered cloud / map

双目图像
  -> TensorRT 立体深度推理
  -> 深度图 / 镜面区域信息
  -> 送入镜面感知模块，参与点级可靠性判断
```

这种双链路汇聚的方式有两个显著优点：

1. LiDAR-IMU 主链路保持闭环完整，可独立工作。
2. 视觉链路以增强器身份接入，既能提升退化场景表现，又不会把整个系统变成一个对视觉强依赖的脆弱结构。

## 5. 配置与外参建模方式

系统在配置上明确区分了两套关系：

1. LiDAR 与 IMU 的融合外参
2. Camera 与 LiDAR 的投影外参

这种拆分非常关键。因为在项目语义里，LiDAR-IMU 关系决定主状态估计的动力学正确性，而 Camera-LiDAR 关系决定视觉证据能否正确投影到点云空间。将这两者分开建模，意味着系统并没有把“多传感器融合”粗糙地看成一个统一黑箱，而是尊重每条观测链路的物理职责。

在工程上，这种分层外参设计带来三点收益：

1. 标定责任边界清晰
2. 模块替换时影响范围更小
3. 针对不同传感器组合更容易复用配置

## 6. 输出与实验闭环

在线系统输出的不只是位姿结果，还包括：

1. odometry
2. path
3. 世界坐标系下注册点云
4. 机体坐标系下点云
5. 地图点云
6. 镜面统计信息
7. 可选 PCD 持久化结果

这类输出设计说明项目不是单纯为了“导航能跑”，而是服务于研究分析。尤其是镜面统计、轨迹记录和地图保存能力，为后续误差分析、可视化和论文指标计算提供了直接基础。

与之对应，`evaluation_tools/` 提供了从轨迹存储、批量统计到结构指标分析的一整套脚本，而 `baselines/` 则提供对照实验环境。这种在线与离线一体化设计，使得项目天然具备以下能力：

1. 记录系统行为
2. 复现实验结果
3. 对比多种方法
4. 支撑定量评测

## 7. 设计哲学

从当前实现可以总结出项目的设计哲学：

### 7.1 以稳定主干为中心，而不是为了融合而融合

项目并没有推翻 FAST-LIO 风格的估计框架，而是保留 LiDAR-IMU 作为主定位骨架。视觉能力的引入服务于“改善观测质量”，而不是把系统改造成一个新的全耦合大型黑箱。

这体现出一种清晰的工程判断：在退化场景里，真正需要增强的是错误观测抑制能力，而不是盲目堆叠更多状态量或更复杂的优化图。

### 7.2 让视觉成为证据增强器，而不是主导者

深度图与镜面 mask 的角色不是输出最终位姿，而是参与点级置信度建模。这说明项目强调“视觉辅助 LiDAR”，而不是“视觉接管 LiDAR”。这样的选择更符合镜面场景的实际问题结构，因为镜面干扰首先破坏的是几何观测可靠性。

### 7.3 优先做弱耦合、低侵入的系统扩展

你们把立体推理节点单独封装、把镜面逻辑集中在独立类里、把实验工具单独放到评测目录，这些都说明架构上追求的是低侵入式扩展。核心好处是：

1. 主算法链路容易保持简洁
2. 子模块可单独替换
3. 研究迭代速度更快
4. 不同成员可并行开发不同子系统

### 7.4 重视实验闭环，而不仅是在线效果

仓库中同时存在基线实现、结果记录、指标统计和批量评估脚本，这代表项目从设计阶段就考虑了论文实验、消融研究和结果复现。换句话说，系统不是只为展示效果而写，而是为“可验证的研究生产”而写。

### 7.5 兼顾实时性、鲁棒性与可扩展性

整体路线不是追求极端复杂的统一模型，而是在实时主链路上增加必要的场景理解能力。LIO 主干负责速度和稳定，视觉增强负责场景鲁棒性，评测工具负责结果闭环。这是一种典型的系统工程平衡思路。

## 8. 模块职责划分

| 模块 | 主要职责 | 在系统中的位置 |
| --- | --- | --- |
| `sentinel_lio` | LiDAR-IMU 紧耦合估计、地图更新、结果发布 | 在线主链路核心 |
| `Preprocess` | 异构 LiDAR 数据适配与扫描组织 | 传感器入口 |
| `ImuProcess` | IMU 去畸变、预测与融合准备 | 状态估计前端 |
| `MirrorSentinel` | 镜面区域理解、点级可信度调制 | 感知增强桥接层 |
| `mirrorsentinel_visual_prior` | DA3 深度和反射 mask prior ROS 封装 | 视觉增强子系统 |
| `evaluation_tools` | 轨迹记录、指标计算、批量评测 | 离线分析层 |
| `baselines` | 对照算法与比较环境 | 研究验证层 |

## 9. 系统价值总结

从系统架构角度看，Sentinel-LIO 的核心价值不只是“在镜面环境下能跑”，而是它采用了一条很有研究与工程平衡感的路线：

1. 以成熟 LIO 主干保证实时稳定。
2. 以视觉深度与镜面理解提升退化场景鲁棒性。
3. 以模块化封装降低多传感器融合复杂度。
4. 以评测和基线体系支撑实验闭环。

这使得项目既适合作为论文系统原型，也适合作为后续继续扩展的研究平台。未来无论是替换更强的视觉模型、加入语义分割、引入更精细的观测筛选策略，还是扩展到更多场景数据集，这套架构都具备比较好的承载能力。

## 10. 结论

Sentinel-LIO 的系统设计不是简单地把多个模块拼接起来，而是围绕“退化场景下如何保护 LIO 主链路的观测质量”这一核心问题展开。其架构体现出三个关键词：

1. 主干稳定
2. 感知增强
3. 实验闭环

从这个意义上说，项目的架构哲学非常明确：以最小必要侵入增强主系统的感知可信度，以模块化组织保证研究迭代效率，以完整评测链路支撑方法论成立。
